# Concepts: Tracing & Observability

## The Problem

An agent processes 1,000 support tickets overnight. In the morning you discover that 80 of them were handled incorrectly. You look at the code — it looks fine. You look at the logs — you see "Request completed." That's it.

Questions you cannot answer:
- Which step failed? Was it retrieval, generation, or post-processing?
- Was it slow? Did timeouts cause retries that cascaded into errors?
- Was it expensive? Did some requests use 10x the expected tokens?
- What was the exact prompt for the failing cases?

Without tracing, you are debugging in the dark.

---

## The Intuition: Distributed Tracing for LLM Calls

If you have worked with microservices, you know distributed tracing: every request gets a trace ID, every service call becomes a span, and a trace visualiser shows you the full execution tree with timing.

Apply the same idea to LLM pipelines:
- Each **LLM call** is a **span** — the atomic unit of observable work
- A **trace** is the sequence of spans for one user request
- Spans record: model name, input tokens, output tokens, latency, cost, errors

The key insight: you don't need a fancy platform to start. You can capture these metrics manually with a simple decorator or context manager and log them to a file or database.

---

## How It Works

### 1. Span — One LLM Call

A span captures everything about a single LLM call:

| Field | Type | Example |
|-------|------|---------|
| `name` | str | "retrieval_reranker" |
| `model` | str | "claude-3-haiku-20240307" |
| `input_tokens` | int | 1200 |
| `output_tokens` | int | 340 |
| `latency_ms` | float | 842.3 |
| `error` | str \| None | None |

### 2. Trace — One User Request

A trace is a list of spans for a single end-to-end request. From a trace you can compute:
- **Total tokens** = sum of all input + output tokens across all spans
- **Total latency** = sum of all span latencies (for sequential pipelines)
- **Total cost** = sum of (input_tokens × input_price + output_tokens × output_price) for each span

### 3. Key Metrics

| Metric | Why It Matters |
|--------|---------------|
| **Latency per span** | Identifies bottleneck steps |
| **Total tokens per trace** | Predicts cost per request |
| **Cost per trace** | Enables billing and budget alerts |
| **Error rate** | Measures reliability per step |
| **p95 latency** | SLA monitoring |

### 4. Observability Tools

| Tool | What It Does |
|------|-------------|
| **LangSmith** | LangChain's built-in tracing — captures prompts, responses, latencies, token counts |
| **LangFuse** | Open-source LLM observability platform — traces, evaluations, cost dashboards |
| **Weights & Biases** | ML experiment tracking extended to LLM pipelines |
| **OpenTelemetry** | Vendor-neutral standard for traces, metrics, logs — integrates with Datadog, Jaeger etc. |

### 5. Manual Tracing: Decorator + Context Manager Pattern

For pipelines where you own the code, a simple decorator is often sufficient:

```python
import time
import functools

def trace_span(name: str):
    def decorator(fn):
        @functools.wraps(fn)
        def wrapper(*args, **kwargs):
            start = time.time()
            try:
                result = fn(*args, **kwargs)
                latency_ms = (time.time() - start) * 1000
                print(f"[TRACE] {name} completed in {latency_ms:.1f}ms")
                return result
            except Exception as e:
                latency_ms = (time.time() - start) * 1000
                print(f"[TRACE] {name} FAILED in {latency_ms:.1f}ms: {e}")
                raise
        return wrapper
    return decorator
```

Key structural rules:
- **`trace_id`** is shared by all spans in the same request — it is the correlation key
- **`parent_span_id: null`** marks the root span

- **`attributes`** are free-form key-value pairs; use namespaced keys (`llm.*`, `db.*`, `tool.*`) for consistency
- **`status`** is either `ok`, `error`, or `unset`

## Instrumenting Your Agent — Code

### The Tracer Class

A minimal `Tracer` wraps the context manager pattern so spans automatically close and record errors even if exceptions bubble up.

```python
import time
import uuid
import functools
from contextlib import contextmanager
from dataclasses import dataclass, field
from typing import Optional, Any

@dataclass
class Span:
    trace_id: str
    span_id: str
    name: str
    parent_span_id: Optional[str] = None
    start_time: float = field(default_factory=time.time)
    end_time: Optional[float] = None
    status: str = "unset"
    attributes: dict = field(default_factory=dict)
    exception: Optional[Exception] = None

    def set_attribute(self, key: str, value: Any) -> None:
        self.attributes[key] = value

    def record_exception(self, exc: Exception) -> None:
        self.exception = exc
        self.attributes["exception.type"] = type(exc).__name__
        self.attributes["exception.message"] = str(exc)
        self.status = "error"

    def finish(self) -> None:
        self.end_time = time.time()
        if self.status == "unset":
            self.status = "ok"


class Tracer:
    def __init__(self, exporter=None):
        self._active_trace_id: Optional[str] = None
        self._active_span_id: Optional[str] = None
        self._exporter = exporter  # callable(span) — e.g. write to DB or send to Langfuse

    @contextmanager
    def start_span(self, name: str, trace_id: Optional[str] = None):
        # Reuse an existing trace_id or start a new trace
        tid = trace_id or self._active_trace_id or uuid.uuid4().hex
        sid = uuid.uuid4().hex
        parent_sid = self._active_span_id

        span = Span(
            trace_id=tid,
            span_id=sid,
            name=name,
            parent_span_id=parent_sid,
        )

        # Push context
        prev_trace_id = self._active_trace_id
        prev_span_id = self._active_span_id
        self._active_trace_id = tid
        self._active_span_id = sid

        try:
            yield span
        except Exception as e:
            span.record_exception(e)
            raise
        finally:
            span.finish()
            # Pop context
            self._active_trace_id = prev_trace_id
            self._active_span_id = prev_span_id
            if self._exporter:
                self._exporter(span)


# Module-level tracer — import this in your agent code
tracer = Tracer()
```

### Adding Attributes and Recording Errors

```python
import anthropic

client = anthropic.Anthropic()

def call_llm(prompt: str, model: str = "claude-3-haiku-20240307") -> str:
    with tracer.start_span("llm_generate") as span:
        span.set_attribute("llm.model", model)
        span.set_attribute("llm.prompt_length", len(prompt))
        try:
            response = client.messages.create(
                model=model,
                max_tokens=512,
                messages=[{"role": "user", "content": prompt}],
            )
            span.set_attribute("llm.input_tokens", response.usage.input_tokens)
            span.set_attribute("llm.output_tokens", response.usage.output_tokens)
            return response.content[0].text
        except Exception as e:
            span.record_exception(e)   # records type, message, sets status=error
            raise
```

### Decorator for Automatic Span Creation

For functions you want traced without changing their bodies, use a decorator:

```python
import functools

def traced(name: Optional[str] = None):
    """Decorator that wraps a function in a span. Uses the function name if no name given."""
    def decorator(fn):
        span_name = name or fn.__name__

        @functools.wraps(fn)
        def wrapper(*args, **kwargs):
            with tracer.start_span(span_name) as span:
                try:
                    return fn(*args, **kwargs)
                except Exception as e:
                    span.record_exception(e)
                    raise
        return wrapper
    return decorator


# Usage — no changes to the function body needed
@traced("embed_query")
def embed(text: str) -> list:
    # ... call embedding API ...
    pass

@traced("vector_search")
def search(embedding: list, top_k: int = 5) -> list:
    # ... call vector DB ...
    pass
```

### Wiring the Full Agent

```python
def run_rag_agent(user_query: str) -> str:
    # The outermost span becomes the root span (parent_span_id = null)
    with tracer.start_span("rag_agent_request") as root:
        root.set_attribute("user_query_length", len(user_query))

        # Child spans nest automatically via the tracer's context stack
        query_embedding = embed(user_query)
        chunks = search(query_embedding)
        answer = call_llm(build_prompt(user_query, chunks))

        root.set_attribute("answer_length", len(answer))
        return answer
```

## The Four Golden Signals for LLM Apps

The classic SRE four golden signals (latency, traffic, errors, saturation) map cleanly onto LLM-specific concerns. These are the four metrics you should alert on from day one.

| Signal | What to Measure | Alert Threshold | Why It Matters |
|--------|----------------|-----------------|----------------|
| **Latency** | p95 end-to-end response time | &gt; 5 seconds | Users abandon slow responses; p95 catches long-tail outliers that mean averages miss |
| **Error rate** | % of requests with `status = error` | &gt; 1% | Above 1% users notice; LLM APIs can throw rate-limit and context-length errors silently |
| **Token usage** | Tokens per request (cost proxy) | &gt; 150% of baseline | Token spikes signal runaway context accumulation or prompt injection; they also drive cost spikes |
| **Quality** | Eval score (LLM-as-judge pass rate) | &lt; 80% pass rate | The one signal unique to AI — response quality can degrade without any errors or latency change |

### Why Each Threshold

**Latency &gt; 5s:** Frontier models typically respond in 1–3 seconds for short completions. A p95 above 5 seconds usually means a stuck tool call, a rate-limit retry loop, or runaway context.

**Error rate &gt; 1%:** At 1,000 requests/hour, 1% = 10 failed requests per hour. Most teams find this is the practical threshold where user-visible failures become a support burden.

**Token usage &gt; 150% of baseline:** Establish a per-endpoint baseline in your first week of production. A 50% spike over that baseline is an early warning — investigate before the billing alarm fires.

**Quality &lt; 80% pass rate:** For an LLM-as-judge eval suite, 80% is a reasonable minimum bar for most use cases. Below this, the model is failing on more than 1 in 5 requests by a human-quality standard.

### Implementing the Quality Signal

```python
# Run your LLM-as-judge eval on a sample of live traffic each hour
def compute_quality_signal(recent_traces: list[dict]) -> float:
    """Returns pass rate (0.0 to 1.0) for recent traces."""
    if not recent_traces:
        return 1.0

    scores = []
    for trace in recent_traces:
        score = llm_judge_score(
            prompt=trace["user_query"],
            response=trace["answer"],
        )
        scores.append(1 if score >= 4 else 0)  # 4/5 = pass

    return sum(scores) / len(scores)
```

## OpenTelemetry vs. LLM-Specific Tools

### OpenTelemetry: The Universal Standard

[OpenTelemetry](https://opentelemetry.io/) (OTel) is the CNCF-standard SDK for producing spans, metrics, and logs in any language. It gives you:
- A vendor-neutral wire format (OTLP)
- Exporters to Datadog, Jaeger, Grafana Tempo, Honeycomb, and dozens more
- Auto-instrumentation for HTTP libraries, databases, and frameworks

**What OTel does not give you:** LLM-specific semantics. There is no built-in concept of `llm.input_tokens`, `llm.model`, prompt text, or eval scores. You can add these as custom span attributes, but there is no standard schema — every team invents their own.

### LLM-Specific Tools: Langfuse, Phoenix, Arize

These platforms extend the OTel model with first-class LLM semantics:

| Feature | OpenTelemetry | Langfuse | Phoenix (Arize) |
|---------|--------------|----------|-----------------|
| Spans & traces | Yes | Yes | Yes |
| Custom attributes | Yes (manual) | Yes | Yes |
| Prompt / completion capture | No | Yes | Yes |
| Token count + cost dashboards | No | Yes | Yes |
| Eval scores (LLM-as-judge) | No | Yes | Yes |
| Dataset management | No | Yes | Yes |
| Open-source | Yes | Yes | Yes (Phoenix) |

**Rule of thumb:** if you already run Datadog or Grafana, start with OTel and add LLM attributes manually. If you are starting fresh or want eval scoring out of the box, use Langfuse or Phoenix.

### Exporting to Langfuse — 3 Lines of Code

```python
# pip install langfuse
from langfuse import Langfuse

langfuse = Langfuse(
    public_key="pk-lf-...",
    secret_key="sk-lf-...",
    host="https://cloud.langfuse.com",   # or your self-hosted URL
)

# Replace the Tracer exporter with a Langfuse flush
def export_to_langfuse(span: Span) -> None:
    langfuse.trace(
        id=span.trace_id,
        name=span.name,
        input=span.attributes.get("llm.prompt"),
        output=span.attributes.get("llm.completion"),
        metadata=span.attributes,
    )

tracer = Tracer(exporter=export_to_langfuse)
```

After that, every span your agent produces flows into the Langfuse UI with token counts, latency, and cost — no other changes to your agent code.

## Key Terms

| Term | Definition |
|------|-----------|
| **Trace** | The complete record of all operations for a single user request |
| **Span** | A single unit of work within a trace — one LLM call, one tool call, one retrieval |
| **Latency** | Time taken for a span or trace to complete, measured in milliseconds |
| **Tokens** | Input and output token counts — the primary driver of LLM API cost |
| **Cost** | USD cost calculated from token counts and per-model pricing |
| **Observability** | The ability to understand a system's internal state from its external outputs |
| **LangFuse** | An open-source LLM observability platform that captures traces, scores, and dashboards |
| **Instrumentation** | Adding tracing/logging code to capture runtime behaviour |
| **OpenTelemetry** | Vendor-neutral CNCF standard for traces, metrics, and logs |
| **Four Golden Signals** | Latency, error rate, token usage, and quality — the four metrics to alert on for LLM apps |

## Interview Angle

**"How would you debug a production AI pipeline where some requests are slow or expensive?"**

Four steps:
1. **Add tracing**: wrap every LLM call with a span that records model, tokens, and latency
2. **Aggregate by trace**: for each user request, sum tokens and latency across all spans to get a request-level view
3. **Identify outliers**: sort traces by total cost or latency — the worst 5% usually reveal the pattern
4. **Drill into spans**: for expensive/slow traces, look at which span is the culprit — is it retrieval, generation, or a downstream tool call?

In production, use LangFuse or LangSmith to get this automatically. For internal tools, a simple span decorator writing to Postgres or S3 is often enough.

## Common Mistakes

| Mistake | What Goes Wrong | Fix |
|---------|----------------|-----|
| Logging only errors | You know it failed but not why or how long it took | Log every call: latency, tokens, model, status |
| Not capturing input/output tokens | Cannot calculate cost or diagnose runaway token usage | Always record `usage.input_tokens` and `usage.output_tokens` from the API response |
| Flat logging (no trace grouping) | Individual log lines but no way to correlate them to a single request | Group spans under a trace ID (UUID per request) |
| Capturing in production but not staging | Surprises in production that staging would have caught | Instrument all environments — latency and cost surprises are common in dev too |
| No quality signal | Error rate is 0% but users are unhappy | Add an LLM-as-judge pass rate metric sampled from live traffic |
