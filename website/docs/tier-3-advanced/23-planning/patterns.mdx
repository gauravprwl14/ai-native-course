# Patterns: Planning & Task Decomposition

---

## Pattern 1: Simple Sequential Planner

The simplest form: LLM generates a numbered list of steps, you execute them in order.

```python
import json
from utils import get_anthropic_client

client = get_anthropic_client()
MODEL = "claude-3-haiku-20240307"

PLAN_PROMPT = """Break the following goal into 3-6 concrete steps.
Return ONLY a JSON array of strings. Example: ["Step 1", "Step 2"]

Goal: {goal}"""

def generate_sequential_plan(goal: str) -> list[str]:
    response = client.messages.create(
        model=MODEL,
        max_tokens=512,
        temperature=0,
        messages=[{"role": "user", "content": PLAN_PROMPT.format(goal=goal)}]
    )
    return json.loads(response.content[0].text)

def execute_sequential_plan(steps: list[str], goal: str) -> list[str]:
    results = []
    context = f"Overall goal: {goal}"
    for i, step in enumerate(steps):
        prompt = f"{context}\n\nCompleted so far: {results}\n\nNow execute step {i+1}: {step}"
        response = client.messages.create(
            model=MODEL,
            max_tokens=512,
            messages=[{"role": "user", "content": prompt}]
        )
        result = response.content[0].text
        results.append(result)
        context += f"\nStep {i+1} result: {result[:200]}"  # keep context bounded
    return results

# Usage
goal = "Write a short summary of the history of the internet"
plan = generate_sequential_plan(goal)
results = execute_sequential_plan(plan, goal)
```

**When to use:** Simple pipelines where each step feeds the next and there are no branches.

## Pattern 2: Plan with Dependencies

A more powerful plan format that uses JSON task objects with `depends_on` fields. This lets you detect which tasks can run in parallel and which must wait.

```python
import json
from dataclasses import dataclass

PLAN_PROMPT = """Break down the goal into 3-7 tasks.
Return ONLY a JSON array with this schema:
[{{"id": "task-1", "description": "...", "depends_on": []}}]

Dependency rules:
- A task in depends_on must appear earlier in the array
- Tasks with empty depends_on can run first (or in parallel)

Goal: {goal}"""

@dataclass
class TaskResult:
    task_id: str
    success: bool
    output: str
    error: str = None

def generate_dependency_plan(goal: str) -> list[dict]:
    response = client.messages.create(
        model=MODEL,
        max_tokens=1024,
        temperature=0,
        messages=[{"role": "user", "content": PLAN_PROMPT.format(goal=goal)}]
    )
    return json.loads(response.content[0].text)

def execute_with_dependencies(plan: list[dict]) -> dict[str, TaskResult]:
    results: dict[str, TaskResult] = {}

    for task in plan:
        # Wait for all dependencies to complete
        dep_context = ""
        for dep_id in task["depends_on"]:
            if dep_id in results and results[dep_id].success:
                dep_context += f"\n{dep_id} output: {results[dep_id].output[:300]}"
            else:
                # Dependency failed — skip this task
                results[task["id"]] = TaskResult(
                    task_id=task["id"],
                    success=False,
                    output="",
                    error=f"Dependency {dep_id} failed or not completed"
                )
                continue

        prompt = f"Execute this task: {task['description']}\nContext from prior tasks:{dep_context}"
        response = client.messages.create(
            model=MODEL,
            max_tokens=512,
            messages=[{"role": "user", "content": prompt}]
        )
        results[task["id"]] = TaskResult(
            task_id=task["id"],
            success=True,
            output=response.content[0].text
        )

    return results
```

**When to use:** Multi-step pipelines where some tasks are independent (can share a parallel pool) and others must sequence.

## Pattern 3: Replanning on Failure

When a task fails, send the failure details and completed results back to the planner for a revised plan.

```python
REPLAN_PROMPT = """A task in the plan failed. Generate a revised plan for the remaining tasks.

Original goal: {goal}

Completed tasks (do NOT repeat these):
{completed}

Failed task:
  ID: {failed_id}
  Description: {failed_desc}
  Error: {error}

Return ONLY a JSON array of remaining tasks with the same schema:
[{{"id": "task-N", "description": "...", "depends_on": []}}]
Start IDs from task-{next_id}."""

def execute_with_replanning(
    goal: str,
    plan: list[dict],
    max_replan_attempts: int = 2
) -> list[TaskResult]:
    results: dict[str, TaskResult] = {}
    remaining_plan = plan[:]
    replan_count = 0

    while remaining_plan:
        task = remaining_plan.pop(0)

        # Build context from completed results
        context = "\n".join(
            f"{r.task_id}: {r.output[:200]}"
            for r in results.values() if r.success
        )
        prompt = f"Execute: {task['description']}\nContext: {context}"

        try:
            response = client.messages.create(
                model=MODEL,
                max_tokens=512,
                messages=[{"role": "user", "content": prompt}]
            )
            results[task["id"]] = TaskResult(
                task_id=task["id"],
                success=True,
                output=response.content[0].text
            )

        except Exception as e:
            results[task["id"]] = TaskResult(
                task_id=task["id"],
                success=False,
                output="",
                error=str(e)
            )

            if replan_count >= max_replan_attempts:
                break  # Give up after too many replan attempts

            # Ask LLM to replan remaining tasks
            completed_summary = "\n".join(
                f"- {r.task_id}: {r.output[:100]}"
                for r in results.values() if r.success
            )
            next_id = len(results) + 1
            replan_resp = client.messages.create(
                model=MODEL,
                max_tokens=1024,
                temperature=0,
                messages=[{
                    "role": "user",
                    "content": REPLAN_PROMPT.format(
                        goal=goal,
                        completed=completed_summary,
                        failed_id=task["id"],
                        failed_desc=task["description"],
                        error=str(e),
                        next_id=next_id
                    )
                }]
            )
            remaining_plan = json.loads(replan_resp.content[0].text)
            replan_count += 1

    return list(results.values())
```

**When to use:** Any production planner. Failures are inevitable — replanning is what separates a robust system from a fragile one.

## Anti-Patterns

| Anti-pattern | What Goes Wrong | Fix |
|--------------|----------------|-----|
| **Over-decomposing** | 50 micro-tasks for a 5-minute job. LLM spends most time planning, not doing. | Keep tasks at 30-second to 5-minute granularity. Aim for 3-7 tasks per plan. |
| **No replanning on failure** | One failed API call or bad tool output kills the entire run silently | Always wrap task execution in try/except and trigger replanning on failure |
| **Ignoring `depends_on`** | Task-3 runs before task-1 output is available; it hallucinates the context | Topologically sort by dependencies before executing |
| **Unbounded context accumulation** | Passing all previous outputs to every task blows the context window | Summarize or truncate prior results; only pass the direct dependencies |
| **Mutable plan mid-execution** | Replanning mid-execution creates inconsistent IDs and ordering | Deep-copy the plan; never mutate the original; track completed IDs separately |
