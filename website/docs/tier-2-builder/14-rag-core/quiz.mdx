Chunks that are too large average out similarity scores across many topics, reducing retrieval precision — you retrieve a chunk that mentions the answer but also many irrelevant things. Chunks that are too small lose context — a single sentence rarely contains enough information for the LLM to answer well. The sweet spot is typically 200–400 words with some overlap."

      explanation: "When retrieved chunks are below a similarity threshold, injecting them anyway causes the LLM to hallucinate answers from irrelevant context — often the most dangerous type of hallucination because it looks grounded. The correct pattern is to check similarity scores, and if none exceed a threshold (typically 0.70), return a clear fallback message indicating no relevant documents were found. This is more honest and more useful than a confident wrong answer."

      explanation: "The key advantage of RAG over fine-tuning is update speed. Fine-tuning bakes knowledge into model weights — adding a new document requires retraining, which takes hours or days and significant compute cost. With RAG, you just embed the new document and add it to the vector store — the knowledge is available instantly on the next query. RAG also provides citation ability (which document was used), which fine-tuned models cannot provide. The tradeoff: fine-tuning is better for internalising style, tone, and reasoning patterns; RAG is better for up-to-date, citable factual knowledge."
