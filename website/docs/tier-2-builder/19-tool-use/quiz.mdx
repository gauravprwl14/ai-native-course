      correct: 1,
      explanation: "Crashing the program on tool errors makes your agent fragile and unrecoverable. Instead, catch the exception and return a descriptive error string as the tool_result content. The model then has the error in its context and can respond intelligently — for example, it might tell the user the tool is unavailable, suggest an alternative, or ask for clarification. Graceful error handling is a core requirement for production agents."

      ],
      correct: 1,
      explanation: "LLMs are stateless — they have no memory between API calls. The only state they have is the messages array you pass each time. When the model requests a tool and you execute it, you must append both the model's tool_use response and your tool_result to the messages array before the next API call. Without this, the model has no way to know what the tool returned and cannot generate a coherent answer."

After a set number of iterations (typically 10), you break the loop and return a fallback message. This is not a hypothetical concern — models can get stuck in loops when a tool keeps returning errors or when the task is ambiguous. Setting temperature=0 makes outputs deterministic but doesn't prevent loops; it may even make them more reliable (and thus more dangerous without a guard)."

Tool use calls external functions — APIs, calculators, databases — and returns their output as tool results. A sophisticated agent might use both: RAG to look up relevant documents and tool use to fetch live data or perform calculations. Understanding when to use each is a key architectural skill."
